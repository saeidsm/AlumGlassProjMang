خلاصه اجرایی

پروژه Alumglass-IR یک سیستم مدیریت پروژه‌های مهندسی نما با ۳۵۶ فایل PHP و حدود ۱۳۷٬۰۰۰ خط کد (بدون وابستگی‌ها) است. سیستم شامل دو زیرپروژه اصلی (ghom و pardis) با قابلیت‌هایی مانند بازرسی، گزارش روزانه، مدیریت انبار، تقویم، مکاتبات و تلگرام است.

بررسی انجام‌شده نشان می‌دهد پروژه آسیب‌پذیری‌های امنیتی بحرانی متعدد، مشکلات جدی معماری و نگهداری، و نقاط بهبود مهم UX/UI دارد. از جمله: ۲۱۱ کوئری SQL بدون Prepared Statement، توکن تلگرام هاردکد شده، فایل info.php با phpinfo() در production، فایل دیتابیس SQL در مسیر عمومی، و ۳۴ فایل کپی/قدیمی بلااستفاده.

یافته‌های امنیتی

۱۸ مورد
# شرح آسیب‌پذیری شدت فایل / محل فاز راهکار پیشنهادی
S01 SQL Injection — کوئری‌های بدون Prepared Statement
۲۱۱ مورد ->query() با متغیر مستقیم در رشته SQL یافت شد. مهاجم می‌تواند دیتابیس را تخریب/استخراج کند.		 بحرانی pardis/daily_report_form_ps.php, pardis/daily_report_print_ps.php, pardis/weekly_report_ps.php, ghom/workshop_report.php و ۲۰+ فایل دیگر فاز ۰ تمام کوئری‌ها را با prepare() و execute() بازنویسی کنید. هرگز متغیر در رشته SQL قرار ندهید.
S02 توکن ربات تلگرام هاردکد شده
توکن API تلگرام به‌صورت plaintext در کد منبع قرار دارد.		 بحرانی telegram_webhook.php:4, test_webhook.php:6 فاز ۰ توکن را فوراً revoke کنید، از Environment Variable یا فایل .env خارج از document root استفاده کنید.
S03 فایل info.php حاوی phpinfo()
اطلاعات کامل سرور (مسیرها، نسخه‌ها، تنظیمات) در دسترس عموم.		 بحرانی info.php فاز ۰ فایل را فوراً حذف کنید. اطلاعات phpinfo فقط در CLI و development مجاز است.
S04 فایل دامپ دیتابیس در document root
ساختار کامل جداول، داده‌ها و اطلاعات حساس در دسترس عمومی.		 بحرانی localhost.sql.txt (200KB) فاز ۰ فایل را فوراً حذف کنید. بکاپ‌ها هرگز نباید در مسیر وب قرار گیرند.
S05 کلید رمز Cron تلگرام هاردکد
Secret key به‌صورت plaintext در کد قرار دارد.		 بحرانی pardis/telegram_cron.php:11 فاز ۰ کلید را از فایل .env بخوانید و کلید فعلی را عوض کنید.
S06 فایل‌های لاگ در مسیر عمومی
لاگ‌های debug شامل اطلاعات حساس در document root.		 بحرانی ghom/api/save_inspection_debug.log, pardis/api/logs/*.log فاز ۰ لاگ‌ها را خارج از document root منتقل کنید یا با .htaccess مسدود کنید.
S07 XSS (Cross-Site Scripting)
متغیرهای $_GET بدون escape مستقیم در HTML درج شده‌اند.		 بحرانی pardis/daily_reports_dashboard_ps.php:378,383,387 و فایل‌های مشابه فاز ۰ تمام خروجی‌ها را با htmlspecialchars(..., ENT_QUOTES, 'UTF-8') فیلتر کنید.
S08 عدم بررسی CSRF در اکثر APIها
بیش از ۲۰ API فاقد توکن CSRF هستند و در برابر حملات CSRF آسیب‌پذیرند.		 بحرانی ghom/api/*.php, pardis/api/*.php فاز ۱ یک middleware مرکزی CSRF ایجاد کنید و تمام POST endpointها را ملزم به بررسی کنید.
S09 عدم بررسی سطح دسترسی (Authorization) در APIها
بسیاری از APIها فقط session check دارند اما Role/Permission بررسی نمی‌کنند.		 بحرانی ghom/api/create_permit.php, ghom/api/confirm_panels_opened.php و ۱۵+ فایل دیگر فاز ۱ یک Authorization middleware مرکزی اضافه کنید. هر endpoint باید role مجاز را بررسی کند.
S10 آپلود فایل بدون اعتبارسنجی پسوند و نوع
در بسیاری از uploaderها نوع فایل بررسی نمی‌شود.		 بحرانی ghom/api/upload_signed_permit.php, ghom/api/save_logo_settings.php فاز ۱ Whitelist پسوندها، بررسی MIME type سمت سرور، و ذخیره فایل‌ها خارج از document root.
S11 فایل‌های Debug/Test در production
فایل‌های تست و دیباگ فعال با display_errors.		 بالا ghom/debug_test.php, pardis/final_test.php, pardis/test_*.php فاز ۰ تمام فایل‌های تست/دیباگ را از production حذف کنید.
S12 عدم هدر Content-Security-Policy
هیچ CSP header تنظیم نشده و XSS injection آسان‌تر است.		 بالا سراسری فاز ۱ CSP header مناسب در فایل header مشترک اضافه شود.
S13 عدم اجبار HTTPS
هیچ redirect از HTTP به HTTPS وجود ندارد.		 بالا سراسری فاز ۱ HSTS header و redirect اجباری HTTP→HTTPS در .htaccess یا bootstrap.
S14 Path Traversal — readfile() با ورودی ناکافی
برخی endpointها فایل را مستقیماً serve می‌کنند.		 بالا ghom/api/view_permit_details.php, pardis/download_packing_template.php فاز ۱ استفاده از realpath() و بررسی اینکه مسیر در دایرکتوری مجاز قرار دارد.
S15 نبود Rate Limiting روی APIها
APIهای عملیاتی فاقد محدودیت نرخ درخواست هستند.		 متوسط تمام فایل‌های api/ فاز ۲ Rate limiter مرکزی (Redis/Memcached-based) یا middleware PHP اضافه شود.
S16 Session Fixation احتمالی
برخی APIها session_start مستقل فراخوانی نمی‌کنند.		 متوسط API endpoints متعدد فاز ۱ همه APIها از طریق bootstrap.php و secureSession() مشترک وارد شوند.
S17 عدم Sanitize ورودی‌ها
اکثر فایل‌ها از filter_var / filter_input استفاده نمی‌کنند.		 بالا سراسری (فقط ۱۰ فایل از filter استفاده می‌کنند) فاز ۱ Input validation layer مرکزی ایجاد شود.
S18 Error Disclosure
برخی فایل‌ها display_errors = 1 دارند.		 بالا ghom/debug_test.php:3 و فایل‌های commented-out فاز ۰ display_errors = Off در php.ini production. لاگ به فایل.

کارایی و سرعت

۱۰ مورد
# شرح مشکل شدت فایل / محل فاز راهکار پیشنهادی
P01 فایل‌های Monolithic بسیار بزرگ
۳۹ فایل بالای 50KB و ۳ فایل بالای 100KB. فایل packing_list_viewer.php با ۳٬۱۸۵ خط! HTML+CSS+JS+PHP همه در یک فایل.		 بالا ۳۹ فایل — به‌خصوص pardis/daily_reports.php (143KB), pardis/packing_list_viewer.php (125KB) فاز ۲ جداسازی CSS/JS به فایل‌های مستقل. شکستن فایل‌ها به کامپوننت‌های کوچکتر. استفاده از Template Engine.
P02 Inline CSS/JS حجیم
۱۴۷ بلوک <style> و ۱۵۰ بلوک <script> inline. هیچ caching مرورگر امکان‌پذیر نیست.		 بالا تقریباً تمام فایل‌های PHP فاز ۲ CSS و JS را به فایل‌های خارجی منتقل کنید تا مرورگر بتواند cache کند.
P03 عدم Pagination در لیست‌ها
بسیاری از داشبوردها تمام رکوردها را یکجا بارگذاری می‌کنند.		 بالا pardis/daily_reports.php, ghom/reports.php و مشابه فاز ۱ Pagination سمت سرور (LIMIT/OFFSET) و lazy loading اضافه شود.
P04 N+1 Query Pattern
در حلقه‌ها برای هر ردیف یک کوئری جداگانه زده می‌شود.		 بالا pardis/daily_reports_dashboard_ps.php:431, pardis/daily_report_form_ps.php فاز ۲ از JOIN استفاده کنید یا داده‌ها را با یک کوئری batch بارگذاری کنید.
P05 عدم HTTP Caching
هیچ Cache-Control / ETag برای صفحات استاتیک تنظیم نشده.		 متوسط سراسری فاز ۲ Cache-Control header برای asset‌های استاتیک. Versioning در URL فایل‌ها.
P06 بارگذاری CDN بدون async/defer
اسکریپت‌های CDN مانع رندر صفحه می‌شوند.		 متوسط فایل‌های header متعدد فاز ۲ اسکریپت‌ها با defer یا async بارگذاری شوند. CSS Critical Inline.
P07 ۱۵ فایل Header/Footer تکراری
هر زیرپروژه ۴-۵ نسخه header مختلف دارد با کد بسیار مشابه.		 بالا ۱۵ فایل header/footer فاز ۲ یک header مشترک responsive با شرط‌بندی ایجاد شود.
P08 عدم Asset Bundling/Minification
هیچ build pipeline برای فشرده‌سازی CSS/JS وجود ندارد.		 متوسط سراسری فاز ۳ استفاده از Webpack/Vite برای bundle و minify کردن assetها.
P09 عدم Database Connection Pooling
هر درخواست اتصال جدیدی ایجاد می‌کند.		 متوسط سراسری فاز ۳ استفاده از persistent connection در PDO یا ProxySQL.
P10 عدم Gzip/Brotli Compression
خروجی HTML/JSON فشرده نمی‌شود.		 متوسط سراسری فاز ۲ mod_deflate یا mod_brotli در Apache فعال شود.

کیفیت کد و معماری

۸ مورد
# شرح مشکل شدت محل فاز راهکار پیشنهادی
Q01 ۳۴ فایل کپی/قدیمی (Dead Code)
فایل‌هایی مانند * copy.php، *old.php، Copy of *.php در production وجود دارند. ریسک امنیتی و سردرگمی.		 بالا ۳۴ فایل در ghom/ و pardis/ فاز ۰ تمام فایل‌های کپی/قدیمی را حذف کنید. از Git برای نسخه‌بندی استفاده کنید.
Q02 عدم استفاده از Version Control (ظاهراً)
وجود فایل‌های «copy» و «old» نشان‌دهنده عدم استفاده صحیح از Git است.		 بالا سراسری فاز ۰ مهاجرت به Git، تعریف .gitignore مناسب، و workflow مشخص (feature branches).
Q03 تکرار شدید کد (DRY Violation)
منطق مشابه در ghom و pardis تکرار شده. headerها ۱۵ نسخه دارند.		 بالا تمام فایل‌های header، footer، API فاز ۲ کتابخانه مشترک (shared library) و معماری کامپوننتی ایجاد شود.
Q04 نبود معماری مشخص (MVC/Service Layer)
PHP/HTML/CSS/JS/SQL همه در یک فایل مخلوط شده‌اند.		 بالا سراسری فاز ۳ جداسازی تدریجی به MVC: ابتدا Data Access Layer جدا شود، سپس Template و Logic.
Q05 نبود Unit Test / Integration Test
هیچ فایل تست خودکاری وجود ندارد.		 متوسط سراسری فاز ۳ PHPUnit را اضافه کنید. ابتدا تست‌ها را برای عملیات حیاتی بنویسید (login، save inspection).
Q06 نبود Coding Standard
سبک نام‌گذاری ناسازگار، بدون PSR compliance.		 متوسط سراسری فاز ۳ PHP CS Fixer / PHPStan اضافه شود. استاندارد PSR-12 رعایت شود.
Q07 نبود مستندات
هیچ README، API Doc، یا Architecture Doc وجود ندارد.		 متوسط سراسری فاز ۲ README.md، نمودار معماری، و Swagger/OpenAPI برای APIها ایجاد شود.
Q08 عدم Error Logging یکپارچه
لاگ‌ها پراکنده و بدون ساختار مشخص هستند.		 متوسط سراسری فاز ۱ Monolog یا مشابه برای لاگ ساختاریافته. فرمت JSON برای تحلیل.

📌 نکته کلیدی
